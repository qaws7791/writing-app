1. **Server‑First, Client‑Minimal**
   - 상호작용이 필요 없는 UI와 데이터 렌더링은 가능한 한 모두 Server Component로 옮긴다.
   - `"use client"`는 상태/이벤트/브라우저 API가 필요한 최소 컴포넌트에만 선언한다.
2. **Automatic Memoization을 신뢰하되, 구조는 스스로 관리**
   - React 19 컴파일러가 불필요한 `useMemo`, `useCallback`, `React.memo`를 제거하거나 재구성해 주므로, 과도한 수동 메모이제이션에 집착하지 않는다.
   - 대신 **작고 순수한 컴포넌트** 구조를 유지해 컴파일러가 최적화하기 쉬운 코드를 작성한다.
3. **짧은 컴포넌트, 한 가지 책임**
   - 하나의 컴포넌트는 하나의 UI/도메인 책임만 가진다(리스트 렌더링, 폼, 카드, 레이아웃 등).
   - 컴포넌트가 많아지는 것을 두려워하지 말고, 함수를 5줄 이내로 자르는 것처럼 **컴포넌트를 쪼개는 쪽으로 치우친다**.





## 2. 일반 원칙

- 서버 컴포넌트에서는 비동기, 동기 호출이 가능하다
- 클라이언트 컴포넌트에서는 동기 호출 및 Hooks를 사용할 수 있다
- 비동기 API 호출은 커스텀 훅으로 감싸서 사용한다
- 비니지스 로직과 유틸리티 함수는 프레임워크나 UI에 의존성이 없어야 한다
- 의존성 주입을 중점으로 SOLID 원칙을 통해 모든 계층을 분리한다.
- 디렉토리 구조는 도메인(feature) 기반으로 분할하고 데이터 흐름을 반영해야 한다.
- Props는 컴포넌트에게 구성(Configuration)에 대해 알려줘야 합니다
- Custom Hooks는 컴포넌트에게 행동(Internal Behavior)에 대해 알려줘야 합니다.
- 컴포넌트에서 UI가 아닌 모든 로직은 커스텀 훅으로 이동한다
- 인라인 조건문은 컴포넌트나 변수로 추출한다
- 연관된 상태는 하나의 객체로, 독립적인 상태는 별도로 유지한다
- useEffect 하나에는 하나의 부수효과만 담는다







## 3. 프로젝트 구조와 모듈 경계 규칙

> 표준 Feature-Sliced Design(FSD)을 실무에 맞게 경량화한 **4-Layer 아키텍처**로, Nextjs 16 App router 기반 프로젝트에서 **구조적 일관성**, **도메인 격리**, **UI/로직 분리**, **외부 의존성 경계 명확화**를 목표로 합니다.



```
src
 ┣━ 📂 app
 ┣━ 📂 views
 ┣━ 📂 features
 ┣━ 📂 domain
 ┗━ 📂 foundation
```



### 3.1 핵심 원칙

- **도메인 중심 설계**: 비즈니스 규칙의 중심을 `domains/<domain>/model`에 둡니다.
- **경계 기반 설계**: `api`와 같은 외부(서버)와 애플리케이션이 통신하는 경계는 DTO 기반 변환을 통해 안전하게 모델을 보호합니다.
- **React 로직 분리**: `application`은 view-model/mutation/store/form 등 UI orchestration을 담당합니다(단, “데이터 조회 훅”은 만들지 않습니다).
- **TanStack Query 표준화**: 데이터 조회는 항상 `queryOptions` 기반 팩토리를 통해서만 정의/사용합니다.
- **Foundation 분리**: 도메인을 모르는 공용 순수 UI/유틸은 `foundation`에 위치합니다.
- **케밥 케이스 강제**: 모든 디렉토리/파일명은 케밥 케이스로 통일합니다.
- 의존성은 단방향, 아래 방향으로만 흐른다. `app → views → features → domain → foundation`

### 3.2 폴더 구조 규칙

- 규칙 1: UI 계층과 도메인 계층을 분리한다
  - `app`: Nextjs 라우팅
  - `foundation`: 도메인을 모르는 순수 UI 컴포넌트 및 유틸리티(구 shared)
  - `domains`: 비즈니스 로직의 핵심(주제별 격리: user/product/cart 등)
  - `views`: 화면에 대한 composition root. 여러 도메인을 조립해 화면을 완성

### 3.3 레이어 설명

#### `foundation` — 기술 인프라 계층

**역할**

애플리케이션의 기반이 되는 기술적 빌딩 블록을 제공한다. **비즈니스 로직을 전혀 모르며**, 어떤 도메인 개념도 포함하지 않는다. 이 계층은 이론상 다른 프로젝트에 복사해도 그대로 동작해야 한다.

- `api`: 공통 HTTP 클라이언트
- `ui`: 도메인 무관 순수 UI
- `lib`: 범용 유틸리티
  - 아래의 "Lib 디렉토리 올바르게 사용하기" 섹션  참고
- `config`: 환경 변수, 전역 상수 등



#### `domain` — 비즈니스 엔티티 계층

### 역할

애플리케이션의 핵심 데이터 구조와 비즈니스 규칙을 정의한다. 화면에 어떻게 표시되는지(UI)에 대해 전혀 모른다

- model: 데이터 상태, 도메인 로직, 검증 스키마

  - 점 표기법을 사용하여 역할 별로 파일을 나눈다.

  - | 파일             | 역할                                      |
    | ---------------- | ----------------------------------------- |
    | `*.types.ts`     | 도메인 타입                               |
    | `*.schema.ts`    | 검증 라이브러리(zod)를 사용한 스키마 정의 |
    | `*.constants.ts` | 상수 정의                                 |
    | `*.service.ts`   | 비즈니스 로직                             |

- ui: 해당 Entity를 사용하는 UI





**하면 안 되는 것**

- `service.ts`에 HTTP 요청 코드를 넣지 않는다. service는 순수 함수 집합이다.
- React, XState, Zustand 등을 import하지 않는다.
- UI 관련 상태 (로딩, 에러 토스트 등)를 여기서 관리하지 않는다.



#### `features` — 독립적 기능 단위 계층

**역할**

사용자가 경험하는 하나의 독립적인 기능을 캡슐화한다. 상태 머신(XState), UI 상태 저장소(Zustand), 서버와의 통신(api), UI 조각(ui)이 하나의 기능 단위로 묶인다. **이 계층이 앱의 복잡도를 흡수하는 핵심 레이어다.**



- components: 상호작용 UI
- hooks: 비동기 API 호출 래핑
- machines: XState 상태 전이 선언
- stores: Zustand 스토어
- repositories: 해당 기능과 관련된 API 요청
- config: 기능 플래그 등 



**해야 하는 것**

- `index.ts`에 공개할 항목만 명시적으로 re-export한다. 내부 구현은 외부에서 직접 import하지 못하도록 한다.
- feature 간 직접 참조는 금지한다. 



#### `views` — 페이지 조립 계층

**역할**

라우트 하나에 대응하는 페이지를 조립하는 계층이다. `features`의 UI 컴포넌트와 `foundation/ui`의 기본 컴포넌트를 레이아웃에 배치하고, 페이지 수준의 데이터 패칭(서버 컴포넌트)을 담당한다. **비즈니스 로직이나 상태 관리를 직접 소유하지 않는다.**





`app`과 `views`의 역할 분리

| 항목                            | `app/*/page.tsx` | `views/*/index.tsx` |
| ------------------------------- | ---------------- | ------------------- |
| Next.js 라우팅                  | ✅                | ❌                   |
| 메타데이터 (`generateMetadata`) | ✅                | ❌                   |
| 서버 컴포넌트 데이터 패칭       | 위임 가능        | ✅                   |
| 레이아웃 조립                   | ❌                | ✅                   |
| feature 컴포넌트 조합           | ❌                | ✅                   |
| 상태 관리 소유                  | ❌                | ❌                   |

`app/`의 `page.tsx`는 가능한 한 얇게 유지한다. 실질적인 내용은 모두 `views`에 있다.



**하면 안 되는 것**

- 한 view에서 다른 view를 import하는 것은 원칙적으로 피한다. 





#### `app` — Next.js 라우팅 계층

**역할**

Next.js App Router의 파일 시스템 라우팅 규칙을 따르는 진입점 계층이다. 라우팅, 레이아웃, 메타데이터, Provider 조립, API Route Handler를 담당한다. 비즈니스 로직을 포함하지 않는 가장 얇은 계층이어야 한다.





## 4. 데이터/폼 흐름 표준

### 4.1 데이터 가져오기 흐름

- HTTP API: openapi-fetch로 호출 → DTO 수신 → mapper로 domain model 변환
- React에서 데이터 가져오기: `queryKey` 계층 + `queryOptions` 팩토리 정의(단일 진실 소스)
- 페이지 수준에서 가져오기: `queryClient.prefetchQuery` 혹은 API 직접 호출(await)



## 새로운 기능 추가 시 체크리스트

새로운 기능을 추가할 때 아래 순서로 진행하면 계층 경계를 지킬 수 있다.

1. **`domain`부터 시작한다.** 타입과 스키마를 먼저 정의한다.
2. **`foundation`에 필요한 인프라가 있는지 확인한다.** 없으면 추가한다.
3. **`features`에 기능 단위를 생성한다.** 
4. **`views`에서 기능 컴포넌트를 조립한다.** feature의 `index.ts`에서 공개된 항목만 사용한다.
5. **`app`에 라우트를 등록한다.** `page.tsx`는 최대한 얇게 유지한다.

## 참고 자료

### Public API

#### 좋은 Public API의 조건

좋은 Public API는 Slice를 **다른 코드와 통합하기 쉽고, 안정적으로 유지보수**할 수 있게 해줍니다.
이를 위해 다음 세 가지 목표를 충족하는 것이 이상적입니다.

1. **내부 구조 변경에 영향 없음**
   - Slice 내부 폴더 구조를 바꾸더라도, 외부 코드는 그대로 동작해야 합니다.
2. **주요 동작 변경 = API 변경**
   - Slice의 동작이 크게 바뀌어 기존 기대가 깨진다면, Public API도 함께 변경되어야 합니다.
3. **필요한 부분만 노출**
   - Slice 전체 구현을 공개하는 것이 아니라, 외부에서 꼭 필요한 기능만 선별해서 노출합니다.

#### 안 좋은 예: 무분별한 Wildcard Re-export

- **발견 가능성 저하**
  - Public API에서 어떤 기능을 제공하는지 한눈에 파악하기 어렵습니다.
- **내부 구현 노출**
  - 원래 외부에서 알 필요가 없는 내부 코드를 외부에서 직접 사용하게 되고,
    이 코드에 대한 의존성이 생기면 리팩터링이 매우 어려워집니다.

### Lib 디렉토리 올바르게 사용하기

문제점

- 모호한 이름: `utils`나 `helpers`라는 이름은 내부 코드의 본질이나 목적을 명확히 설명하지 못합니다
- **유지보수의 어려움:** 프로젝트가 커지면서 이 디렉토리에 함수가 무분별하게 쌓이게 되고, 결국 구조를 파악하기 힘들어져 중복 코드가 발생할 가능성이 매우 높습니다.

해결책

명확한 목적을 가진 **독립적인 라이브러리** 형태로 관리하세요.

- 명확한 명명: `lib/datetime`처럼 역할이 분명한 이름을 부여하세요.
- **완결성 확보:** 각 라이브러리는 자체적인 문서와 **테스트 코드**를 갖추어야 합니다.
