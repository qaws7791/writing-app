### 1. Apollo GraphQL을 활용한 데이터 및 에러 처리 최적화 (발표자: Henry)

**커스텀 스칼라(Custom Scalars) 처리의 한계와 해결책**
GraphQL 서버에서 Date와 같은 커스텀 스칼라 타입을 전송할 때, 클라이언트는 이를 객체가 아닌 단순 문자열(String) 형태로 전달받게 되어 프론트엔드 단의 데이터 매핑 작업이 복잡해집니다. [[04:11](http://www.youtube.com/watch?v=WxK11RsLqp4&t=251)]

- 해결 방법으로 Apollo Link를 미들웨어처럼 활용하여, 데이터가 클라이언트 측 로직에 도달하기 전에 가로채어 원하는 객체 형태(예: Date 객체)로 역직렬화(Deserialize)하는 방식이 유효합니다. [[05:18](http://www.youtube.com/watch?v=WxK11RsLqp4&t=318)]
- 대안으로 서버 측 스키마를 동적으로 읽어와 클라이언트에서 자동으로 역직렬화를 수행하는 별도의 라이브러리를 도입해 데이터 매핑 부담을 줄이는 접근도 가능합니다. [[09:16](http://www.youtube.com/watch?v=WxK11RsLqp4&t=556)]

**에러(Error)의 재정의와 유니온(Unions) 타입 활용**
기존 REST API 환경에서는 200 OK 이외의 응답을 대부분 에러로 취급하는 경향이 있습니다. 하지만 GraphQL 환경에서는 '리소스 없음', '토큰 만료'와 같은 예상 가능한 예외 상황을 서버의 내부 에러(Internal Server Error)와 완전히 동일하게 취급해서는 안 됩니다. [[12:04](http://www.youtube.com/watch?v=WxK11RsLqp4&t=724)]

- 예상 가능한 예외 상황은 비즈니스 로직 플로우의 일부로 간주해야 합니다. 따라서 유니온(Unions) 타입을 활용하여 '성공 결과'와 '대안적 결과(비즈니스 예외)'를 모두 정상적인 응답 구조로 묶어 클라이언트에 반환하는 것이 바람직합니다. [[13:19](http://www.youtube.com/watch?v=WxK11RsLqp4&t=799)]
- 반면 GraphQL 스키마 표준 내에서 입력 값에 대한 유니온(Input Unions) 지원은 아직 완전하지 않아, 여러 타입 중 하나를 입력받아야 할 때 입력 객체 내에 선택적(Optional) 필드를 여러 개 두는 형태의 설계가 강제되기도 합니다. [[14:33](http://www.youtube.com/watch?v=WxK11RsLqp4&t=873)]

**스키마 유효성 검사와 커스텀 에러 응답 설계의 어려움**
스키마는 문서화 및 기본적인 데이터 타입 유효성 검사에는 훌륭하지만, 복잡한 커스텀 유효성 검사 실패에 대해 유연한 형태의 응답을 제공하는 데는 한계가 있습니다. 서버 로직에서 예외(Exception)를 발생시키면 스펙에 의해 일률적인 에러 객체 형태로 래핑되어 반환되므로 클라이언트가 이를 분석하고 파싱하기 불편합니다. [[18:04](http://www.youtube.com/watch?v=WxK11RsLqp4&t=1084)]

- 이를 우회하기 위해, 발생한 에러를 자체적인 식별 객체로 한 번 더 감싸서 던진 뒤, 상위 리졸버(Resolver)에서 이 식별 객체를 감지하여 유니온 타입에 정의된 커스텀 실패 응답으로 변환해 반환하는 형태의 기법(Hack)이 사용될 수 있습니다. [[20:20](http://www.youtube.com/watch?v=WxK11RsLqp4&t=1220)]

### 2. 확장 가능한 실시간 동기화(Real-time Sync) 아키텍처 구축 (발표자: Tuomas Artman, Linear)

**실시간 동기화 계층의 필요성과 개발 생산성 극대화**
현대적인 협업 애플리케이션에서는 다수의 사용자가 동일한 화면을 보고 조작할 때 상태가 즉각적으로 일치하는 실시간 멀티플레이어 환경이 요구됩니다. 이러한 실시간 동기화 엔진을 애플리케이션의 기반 계층에 구축해 두면, 뷰(View) 단에서 네트워크 호출(REST, GraphQL 등)이나 수동 상태 갱신 로직을 신경 쓸 필요 없이 데이터를 수정하고 저장(Save)하는 코드 한 줄만으로 모든 UI가 자동으로 업데이트되어 엔지니어링 리소스가 극적으로 절감됩니다. [[43:15](http://www.youtube.com/watch?v=WxK11RsLqp4&t=2595)]

**동기화 엔진 구성을 위한 핵심 아키텍처**

- **오브젝트 풀(Object Pool)과 정규화:** 중복 참조나 순환 참조가 발생하는 복잡한 JSON 트리 데이터를 직접 다루는 대신, 모든 데이터를 평면적인 1차원 오브젝트 풀(Object Pool)로 정규화하여 관리합니다. [[49:07](http://www.youtube.com/watch?v=WxK11RsLqp4&t=2947)]
- **데코레이터(Decorators) 기반 동적 관계 매핑:** 데코레이터를 활용하여 일대다, 다대일 등 엔티티 간의 관계를 메타데이터로 선언해 둡니다. 오브젝트 풀에 새로운 데이터가 스트리밍되거나 삭제될 때, 데코레이터 로직이 발동하여 자동으로 객체 간의 참조 그래프(Object Graph)를 올바르게 재구성합니다. [[52:52](http://www.youtube.com/watch?v=WxK11RsLqp4&t=3172)]
- **반응형 상태 관리 체계 (MobX):** 구성된 객체 참조 그래프는 MobX와 같은 반응형 라이브러리에 의해 관찰(Observable) 상태가 됩니다. 데이터가 변경되면 해당 데이터를 구독하고 있던 뷰 컴포넌트만 정확하게 감지하여 리렌더링을 수행합니다. [[46:11](http://www.youtube.com/watch?v=WxK11RsLqp4&t=2771)]

**낙관적 UI(Optimistic UI) 롤백 체계와 트랜잭션**

- 사용자가 데이터를 변경하면 오브젝트 풀은 이를 즉시 뷰에 반영하고, 변경 내역을 '트랜잭션(Transaction)'으로 패키징하여 큐(Queue)에 적재합니다.
- 클라이언트는 서버의 승인을 기다리지 않고 낙관적으로 UI를 먼저 변경한 상태를 유지합니다. 백엔드 처리 후 승인이 이루어지면 해당 트랜잭션을 큐에서 폐기하고, 서버에서 거절(Validation Fail 등)될 경우 트랜잭션에 사전 기록해둔 이전 상태(Previous Value)를 불러와 오브젝트 풀을 롤백시킴으로써 UI를 원상 복구합니다. [[01:02:05](http://www.youtube.com/watch?v=WxK11RsLqp4&t=3725)]

**오프라인 지원(Offline Support) 및 웹소켓 브로드캐스트**

- 브라우저의 IndexedDB와 같은 로컬 데이터베이스에 전체 오브젝트 풀과 현재 대기 중인 트랜잭션 큐를 저장함으로써 완벽한 오프라인 모드를 지원합니다. 네트워크가 단절된 상태에서의 사용자 액션은 모두 로컬 트랜잭션 큐에 쌓이며, 네트워크가 복구되면 큐에 누적된 데이터가 순차적으로 백엔드에 전송됩니다. [[01:06:13](http://www.youtube.com/watch?v=WxK11RsLqp4&t=3973)]
- 서버에서 최종적으로 승인된 데이터 변경 사항은 웹소켓(WebSockets)을 통해 연결된 모든 클라이언트의 Sync Resolver로 즉시 스트리밍되며, 개별 클라이언트의 오브젝트 풀이 갱신되면서 전체 사용자에게 실시간 동기화가 이루어집니다. [[01:04:50](http://www.youtube.com/watch?v=WxK11RsLqp4&t=3890)]
