Linear 동기화 엔진(Sync Engine)의 아키텍처 확장과 성능 최적화 고찰

Linear 애플리케이션의 핵심 기반인 동기화 엔진(Sync Engine)은 클라이언트 중심의 실시간(Real-time) 데이터 처리 아키텍처를 통해 네트워크 지연을 최소화하고 높은 사용자 경험과 엔지니어링 생산성을 제공하도록 설계되었다. 이 시스템은 모든 데이터를 로컬 환경(클라이언트 메모리 및 IndexedDB)에 적재하고 옵티미스틱 업데이트(Optimistic Updates)를 수행함으로써 오프라인 지원과 즉각적인 UI 반응성을 보장한다. 그러나 애플리케이션이 스케일업(Scale-up)하고 대규모 조직 및 데이터 세트를 다루게 되면서 다양한 성능 병목 현상이 발생하였고, 이를 해결하기 위한 다각적인 기술적 진화가 이루어졌다. 본 아티클에서는 Linear 동기화 엔진의 초기 설계부터 대규모 확장성 확보를 위해 도입된 최적화 기법들을 아키텍처 관점에서 상세히 분석한다.

**1. 동기화 엔진의 기초 아키텍처 및 객체 그래프 (Foundation of the Sync Engine)**

Linear의 동기화 엔진은 본질적으로 클라이언트에 전체 데이터 상태를 복제하는 방식에 의존한다. 클라이언트는 초기 구동 시 백엔드의 GraphQL API를 통해 데이터베이스에 접근하여 사용자가 접근 가능한 모든 데이터를 거대한 JSON 형태로 가져온다(Full Bootstrap). 수신된 데이터 객체 풀(Object Pool)은 내부 참조(References)를 통해 관계형 구조를 형성하고, Workspace(Organization)를 루트(Root)로 하는 인메모리 객체 그래프(In-memory Object Graph)를 구성한다.

UI 계층은 MobX를 통해 이 객체 그래프와 바인딩된다. 엔진은 모델 객체의 속성이 변경될 때마다 이를 관찰(Observe)하여 의존성이 있는 컴포넌트만 선택적으로 재렌더링한다. 사용자가 데이터를 변경하면 클라이언트는 네트워크 응답을 기다리지 않고 로컬 메모리를 즉시 수정하여 UI를 업데이트한다(Optimistic Update). 동시에 이 변경 사항은 로컬 디스크의 트랜잭션 큐(Transaction Queue)에 적재되어 백엔드로 비동기 전송된다. 서버가 데이터 충돌 등을 이유로 유효성을 검증하여 트랜잭션을 거부(Reject)하면 로컬 상태는 롤백되며, 승인된 변경 사항은 서버의 Sync Action 테이블에 기록된 후 WebSocket을 통해 다른 모든 클라이언트에게 브로드캐스트된다.

**2. 데이터 델타 동기화(Delta Sync) 메커니즘**

초기 전체 부트스트랩(Full Bootstrap)이 끝난 후, 클라이언트가 백엔드와의 지속적인 데이터 정합성을 유지하기 위해 델타 동기화(Delta Sync)가 사용된다. 각 동기화 이벤트는 순차적으로 증가하는 Sync ID를 가진다. 클라이언트가 오프라인 상태에서 다시 온라인으로 전환되거나 새로고침을 할 때, 마지막으로 로컬에 기록된 Sync ID를 바탕으로 서버에 델타 업데이트를 요청한다. 서버는 해당 ID 이후에 발생한 모든 데이터의 변경분(Delta)을 GraphQL API를 통해 클라이언트에 전달한다. 이 과정 중 WebSocket으로 실시간 수신되는 새로운 업데이트들은 메모리의 별도 큐에 임시 보관되었다가 델타 동기화가 완전히 로컬 데이터베이스에 적용된 즉시 순차적으로 재생(Replay)되어 레이스 컨디션(Race Condition)을 방지한다.

**3. 클라이언트 성능 병목 해결: 지연 로딩과 부분 부트스트랩 (Partial Bootstrap)**

조직의 규모가 커짐에 따라 클라이언트 내 IndexedDB에서 수만 개의 모델 객체를 읽어 들이고 이를 MobX Observable 객체로 인스턴스화하는 과정에서 심각한 클라이언트 로드 속도 저하(Bootstrap Delay)가 발생했다.

이를 해결하기 위해 모든 데이터를 한 번에 메모리에 올리는 대신 지연 로딩(Lazy Loading) 및 부분 부트스트랩(Partial Bootstrap)을 도입했다. 애플리케이션 진입 시 즉시 필요하지 않은 데이터(예: 특정 이슈의 댓글(Comments), 이슈 히스토리 등 수량이 방대한 객체)는 초기 로드 단계에서 배제되었다. 대신 `Lazy Collection`이라는 특수한 구조를 구현하여, 엔지니어가 코드 상에서 해당 컬렉션의 데이터에 접근(Getter 호출)하는 시점에 동적으로 IndexedDB를 조회하고 데이터를 수화(Hydration)하도록 설계했다. 또한 데이터 모델 전체를 사전에 MobX로 감싸는 대신, 특정 모델이 ID를 통해 조회되거나 컬렉션으로 접근될 때에만 한정적으로 직렬화 및 Observable 처리를 수행하도록 최적화함으로써 부트스트랩 소요 시간을 비약적으로 단축시켰다.

**4. 서버 메모리 초과 이슈와 Streaming REST Endpoint 도입**

클라이언트 부트스트랩 속도가 개선되자 병목 현상은 백엔드의 GraphQL API로 이동했다. GraphQL은 구조적 특성상 클라이언트에게 반환할 거대한 JSON 응답을 전송하기 전에 서버 메모리에 전체 응답 데이터를 완전히 구축(Build)해야만 한다. 수십에서 수백 메가바이트에 달하는 데이터 응답을 다수의 클라이언트가 동시에 요청할 경우, 서버 포드(Pod)의 메모리 할당량을 쉽게 초과하여 서비스가 다운되는 현상이 빈번해졌다.

Linear 팀은 대용량 데이터 전송에서 발생하는 GraphQL의 근본적 한계를 극복하기 위해 동기화 전용 내부 'Streaming REST Endpoint'를 신규로 구축했다. 이 엔드포인트는 데이터베이스에서 쿼리 결과를 한 줄씩 읽어 들여(Row-by-Row) 메모리에 누적하지 않고 곧바로 클라이언트 측에 스트리밍 방식으로 전송한다. 클라이언트의 네트워크 대역폭이 좁아 서버의 스트리밍 버퍼가 임계치를 초과할 경우에는, 데이터를 메모리 대신 디스크에 임시로 스풀링(Spooling)하여 서버 메모리 고갈 문제를 원천적으로 차단했다.

**5. 데이터베이스 부하 분산 및 NoSQL 캐시 계층 (MongoDB) 구축**

백엔드 서버의 메모리 문제가 해결된 후에는 수많은 Full Bootstrap 요청에 의한 핵심 관계형 데이터베이스(PostgreSQL)의 CPU 연산 부하가 새로운 병목으로 떠올랐다. 거대한 트리 형태의 복잡한 객체 구조를 처리하기 위해 RDBMS에서 매번 Join 및 대량 쿼리를 수행하는 것은 비효율적이었다. 읽기 복제본(Read Replica)을 활용하는 방안도 시도되었으나, 복제 지연(Replication Lag)에 따른 정합성 제어 비용이 컸다.

이에 대한 해법으로 몽고DB(MongoDB)를 활용한 NoSQL 캐시 계층이 도입되었다. 정기적으로, 혹은 데이터 변경량이 일정 수준을 초과할 때마다 PostgreSQL 내의 데이터 상태를 사전에 완전히 직렬화하여 MongoDB에 단일 문서 덤프(Dump) 형태로 저장한다. 이후 클라이언트의 부트스트랩 요청은 복잡한 쿼리 연산 없이 MongoDB에서 해당 조직의 직렬화된 데이터를 통째로 빠르게 읽어 반환함으로써 주 데이터베이스의 부하를 획기적으로 낮추었다. 덤프 이후의 최신 데이터 변경분은 기존의 Delta Sync 로직을 병행 적용하여 데이터 무결성을 유지한다.

**6. 대규모 확장(Scaling)을 위한 배치 로더(Batch Loader) 메커니즘**

가장 최근에 대두된 문제는 아무리 지연 로딩과 캐싱을 적용하더라도 엔터프라이즈급 조직이 접근할 경우 초기 클라이언트 로드 시 필요한 네트워크 페이로드 크기 자체가 물리적으로 지나치게 크다는 점이었다.

이를 해결하기 위해 클라이언트는 조직, 팀, 사용자 등 핵심 코어 메타데이터만을 다운로드하고, 방대한 '이슈(Issues)'나 '첨부 파일(Attachments)'과 같은 무거운 모델 객체들은 클라이언트 로컬 스토리지에조차 초기에 저장하지 않는 구조로 고도화되었다. 대신 UI 컴포넌트 렌더링 시점에 네트워크 요청을 동적으로 발생시키는 배치 로더(Batch Loader) 아키텍처를 도입했다.

컴포넌트 렌더링 과정에서 여러 Lazy Collection이 데이터를 동시다발적으로 요청하면, 이를 개별적으로 네트워크에 전송하지 않고 약 50ms의 윈도우(Window) 동안 발생한 요청들을 큐잉(Queueing)한다. 이후 백그라운드에서 이 요청들을 분석하여 중복을 제거(Deduplication)하고 최적의 그룹으로 병합(Grouping)하여 단일 네트워크 페이로드로 백엔드에 전송한다.

이 과정에서 TypeScript의 타입 시스템을 정교하게 활용해 비동기 데이터를 `Cached Promise` 형태로 관리한다. 네트워크로부터 데이터가 로드되고 수화(Hydrate)될 때까지 React의 Suspense Boundary를 사용해 UI 렌더링의 흐름을 우아하게 제어한다. 이로써 엔지니어는 데이터가 로컬 디스크에 있는지, 혹은 네트워크로부터 실시간으로 로드되어야 하는지 알 필요 없이 기존의 동기식 객체 접근 API 코드를 그대로 유지하면서 대규모 비동기 데이터 로딩을 효율적이고 안정적으로 처리할 수 있게 되었다.
