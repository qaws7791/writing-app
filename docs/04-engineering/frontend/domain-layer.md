# 프론트엔드 도메인 레이어: 흩어진 비즈니스 로직을 제자리로

## 들어가며

프론트엔드 코드베이스가 일정 규모를 넘어서면, 어느 시점부터 묘한 불편함이 쌓이기 시작한다. 특정 기능을 수정하려고 파일을 열었을 때, 내가 고쳐야 할 로직이 컴포넌트 안에 있는지, 커스텀 훅 안에 있는지, 아니면 API를 호출하는 서비스 함수 어딘가에 섞여 있는지 금방 파악되지 않는 경험. 비슷한 계산이 여러 컴포넌트에서 조금씩 다른 형태로 반복되고 있다는 사실을 나중에야 발견하는 경험. 이런 증상들은 대부분 같은 원인에서 비롯된다. **도메인 로직에 명확한 위치가 없다는 것.**

이 글은 그 문제를 해결하기 위한 하나의 접근법, 즉 프론트엔드 아키텍처 안에 **도메인 레이어(Domain Layer)** 를 명시적으로 도입하는 것의 의미와 가치에 대해 이야기한다.

---

## 지금 코드에는 무슨 일이 벌어지고 있는가

전통적인 프론트엔드 코드 구조에서 로직은 대략 세 곳에 머문다.

- **컴포넌트** — UI 렌더링을 책임지지만, 어느 순간 `if` 분기와 데이터 가공 로직이 뒤섞인다.
- **커스텀 훅** — 상태와 부수 효과를 추상화하지만, 비즈니스 규칙이 함께 흘러들어온다.
- **유스케이스 / 서비스 함수** — API 호출과 데이터 흐름을 조율하지만, 도메인 개념이 중간중간 끼어든다.

문제는 이 중 어느 곳도 도메인 로직의 **적절한 집** 이 아니라는 데 있다. 컴포넌트는 렌더링에 집중해야 하고, 커스텀 훅은 리액티브한 상태 관리에 집중해야 하고, 유스케이스 함수는 흐름 조율에 집중해야 한다. 도메인 로직은 이 모든 레이어를 오가며 스스로의 정체성 없이 떠돌게 된다.

예를 들어 "주문 금액에 따라 할인율이 달라지는 정책"이 있다고 해보자. 이 정책은 장바구니 컴포넌트에서 금액을 표시할 때도 필요하고, 결제 확인 훅에서도 필요하고, 주문 요약 화면에서도 필요하다. 자연스럽게 같은 로직이 세 곳에 복사되거나, 각자의 방식으로 조금씩 다르게 구현된다. 정책이 바뀌면? 세 곳을 모두 찾아야 한다. 그리고 한 곳을 놓치면 버그가 생긴다.

---

## 도메인 로직이란 무엇인가

도메인 레이어를 논하기 전에, 먼저 도메인 로직의 경계를 짚고 넘어갈 필요가 있다.

도메인 로직이란 **애플리케이션이 해결하려는 문제 그 자체에 속한 규칙과 계산** 이다. UI가 어떻게 생겼는지, 서버와 어떻게 통신하는지와는 무관하게, 비즈니스 자체가 가진 논리다.

- 특정 조건을 만족할 때 쿠폰을 적용할 수 있는가?
- 사용자의 역할에 따라 어떤 기능에 접근 가능한가?
- 날짜 범위의 유효성을 어떻게 판단하는가?
- 주문 상태가 어떤 전이(transition)를 허용하는가?

이런 규칙들은 React에 의존하지 않고, fetch에 의존하지 않고, 심지어 브라우저에도 의존하지 않는다. 그것이 핵심이다. **순수하게 JavaScript/TypeScript의 함수와 클래스, 타입으로 표현될 수 있는 것이 도메인 로직이다.**

---

## 도메인 레이어가 하는 일

도메인 레이어는 이 로직들을 한곳에 모아, 나머지 레이어로부터 독립적으로 존재하게 만드는 구조적 결정이다.

```
src/
  domain/
    order/
      order.model.ts       # 주문 도메인 타입과 값 객체
      order.policy.ts      # 주문 관련 비즈니스 규칙
      order.calculator.ts  # 금액 계산 로직
    user/
      user.model.ts
      user.permission.ts
```

이 레이어의 가장 중요한 특징은 **의존 방향** 이다. 도메인 레이어는 컴포넌트도, 훅도, API 클라이언트도 알지 못한다. 외부 세계에 대한 의존성이 없다. 반면 컴포넌트와 훅, 유스케이스는 도메인 레이어를 참조할 수 있다. 의존성이 단방향으로 흐른다.

이것은 단순한 폴더 구조의 문제가 아니다. **"이 코드는 비즈니스 규칙이다"라는 명시적인 선언** 이다.

---

## 무엇이 좋아지는가

### 1. 로직이 찾아야 할 위치가 생긴다

할인 정책을 수정해야 할 때, 어디를 열어야 하는지 알 수 있다. 주문 상태 전이 규칙이 바뀌었을 때, 그 변경이 어디에 집중되어야 하는지 팀 전체가 동의할 수 있다. 코드를 읽는 사람이 "이건 비즈니스 로직이구나"라는 것을 파일 경로만 봐도 직관적으로 알 수 있다.

이것은 팀 커뮤니케이션 비용을 줄이는 효과가 있다. 새로 합류한 팀원이 도메인 규칙을 파악하고 싶을 때, 방대한 컴포넌트 트리를 순회하지 않고 도메인 레이어를 읽는 것만으로 충분해진다.

### 2. 테스트가 쉬워진다

도메인 로직이 React와 분리되어 있다는 것은, **컴포넌트를 렌더링하지 않고도 비즈니스 규칙을 테스트할 수 있다**는 의미다. 할인 계산 함수를 테스트하기 위해 모킹이 필요한 환경을 구성할 필요가 없다. 입력과 출력만 있는 순수 함수이기 때문에, 테스트는 간결하고 빠르며 신뢰할 수 있다.

반대로 말하면, 지금까지 컴포넌트 테스트가 무겁고 부서지기 쉬웠던 이유 중 하나가 도메인 로직이 컴포넌트 안에 들어가 있었기 때문이다. 컴포넌트 테스트는 렌더링 결과와 상호작용을 검증해야지, 비즈니스 규칙의 정확성을 검증하는 자리가 되어서는 안 된다.

### 3. 재사용이 자연스러워진다

같은 도메인 규칙이 여러 곳에서 필요할 때, 이제는 복사하거나 별도로 구현하지 않아도 된다. 도메인 레이어에서 가져다 쓰면 된다. 코드의 단일 진실 원천(single source of truth)이 생기는 것이다.

더 나아가, 동일한 도메인 로직이 웹 클라이언트뿐 아니라 서버 사이드 렌더링 로직이나 Node.js 기반의 BFF(Backend for Frontend)에서도 재사용될 수 있다. 프레임워크에 의존하지 않으므로, 실행 환경이 달라도 같은 코드가 동작한다.

### 4. 도메인 언어가 코드에 반영된다

도메인 레이어를 도입하면, 팀이 사용하는 비즈니스 언어가 코드에 직접 투영된다. `isEligibleForDiscount`, `canTransitionTo`, `calculateFinalPrice` 같은 이름들은 단순한 유틸리티 함수가 아니라, 비즈니스 도메인의 어휘를 코드로 표현한 것이다.

이것은 DDD(Domain-Driven Design)에서 말하는 유비쿼터스 언어(Ubiquitous Language)의 실천과도 맞닿아 있다. 개발자와 기획자, 디자이너가 같은 언어로 문제를 이야기할 때, 그 언어가 코드에도 그대로 살아있다면 맥락의 손실 없이 변경을 추적할 수 있다.

## 흔한 오해와 주의할 점

### "그냥 유틸 폴더 아닌가?"

유틸리티는 범용적인 도구다. 날짜 포맷팅, 문자열 처리, 타입 가드 같은 것들. 도메인 로직은 특정 비즈니스 문제에 속한 규칙이다. 할인 정책, 권한 체계, 주문 흐름 같은 것들. 이 둘은 성격이 다르고, 같은 폴더에 있어서는 안 된다.

유틸 폴더에 비즈니스 규칙이 섞이는 순간, 그 규칙은 다시 제자리를 잃는다.

### "레이어가 하나 더 생기면 복잡해지지 않나?"

레이어가 늘어나면 초기에 익혀야 할 구조가 늘어나는 것은 사실이다. 하지만 레이어의 목적은 복잡성을 추가하는 것이 아니라 **복잡성을 제자리에 정리하는 것**이다. 규모가 작은 프로젝트에서는 필요하지 않을 수도 있다. 그러나 팀의 규모가 커지고, 도메인이 복잡해지고, 코드베이스의 수명이 길어질수록 이 구조적 명확함은 점점 더 큰 가치를 발휘한다.

### "모든 로직을 도메인 레이어로 올려야 하나?"

그렇지 않다. 특정 컴포넌트에만 국한된 UI 상태 로직, 특정 API 응답을 가공하는 변환 로직 등은 도메인 로직이 아닐 수 있다. 도메인 레이어로 올릴 기준은 하나다. **"이 규칙은 UI가 없어도, API가 없어도 여전히 유효한가?"** 그 질문에 예스라고 답할 수 있다면 도메인 레이어에 속한다.
