# 프론트엔드 API 호출 로직의 함수 분리

## 개요

프론트엔드 개발에서 API 호출 로직을 의미 있는 함수 단위로 분리하는 것은, 컴포넌트가 "무엇을 렌더링할 것인가"에만 집중할 수 있도록 **관심사를 명확히 나누는** 설계 원칙이다. 이 기법은 단순한 코드 정리 수준을 넘어, 유지보수성·재사용성·테스트 가능성 전반에 영향을 미치는 아키텍처적 결정이다.

---

## 왜 분리가 필요한가

컴포넌트 내부에 API 호출 코드가 직접 작성될 경우, 프로젝트 규모가 커질수록 다음과 같은 문제가 누적된다.

- **중복 코드 확산** — 동일한 엔드포인트에 대한 호출이 여러 컴포넌트에 흩어져 반복된다
- **변경 비용 증가** — API 스펙이 변경될 때, 수정이 필요한 지점을 모두 추적하기 어렵다
- **UI와 로직의 강결합** — 버튼의 위치를 바꾸려다 네트워크 로직까지 건드려야 하는 상황이 발생한다
- **테스트 단절** — UI와 API 로직이 뒤섞여 있으면, 각각을 독립적으로 검증하기 어렵다.
- **팀 협업 저해** — 어느 컴포넌트가 어떤 API를 호출하는지 파악하기 위해 컴포넌트 내부를 읽어야 한다

---

## 핵심 개념: 관심사의 분리(SoC)

API 호출 로직을 분리하는 근본적인 이론적 토대는 **관심사의 분리(Separation of Concerns)** 원칙이다. 이 원칙은 각 모듈이 단 하나의 책임만 가지도록 설계할 것을 요구한다. 프론트엔드 맥락에서는 다음 세 가지 책임이 뚜렷하게 구분된다.

| 책임 영역   | 담당 주체          | 관심사                          |
| ----------- | ------------------ | ------------------------------- |
| 데이터 요청 | API 함수 모듈      | 엔드포인트, 파라미터, 응답 형태 |
| 상태 관리   | Store / Query Hook | 서버 상태, 로딩·에러 상태       |
| 화면 표시   | UI 컴포넌트        | 레이아웃, 인터랙션, 렌더링      |

컴포넌트는 API 함수를 **호출**할 뿐, 그 내부 동작(HTTP 메서드, 헤더, URL 구성 등)을 알 필요가 없어야 한다.

## 함수 분리의 계층 구조

API 호출 로직은 일반적으로 두 가지 계층으로 나뉜다.

### 기반 통신 계층 (Transport Layer)

HTTP 클라이언트를 추상화하는 레이어로, 공통 헤더 주입·인증 토큰 처리·에러 인터셉트 등 모든 요청에 공통으로 적용되는 로직을 담당한다. 이 계층은 특정 도메인 지식을 갖지 않으며, 순수하게 네트워크 통신 인프라 역할만 수행한다.

### 도메인 API 계층 (Domain API Layer)

비즈니스 도메인별로 구성된 함수 집합이다. 예를 들어 사용자(user), 상품(product), 주문(order) 등의 단위로 모듈이 분리되며, 각 함수는 "어떤 리소스를, 어떤 의도로 요청하는가"를 이름으로 명확히 표현한다. 이 함수들은 기반 통신 계층을 사용하되, 컴포넌트가 직접 참조하는 공개 인터페이스가 된다.

---

## 함수 명명의 중요성

함수 분리의 핵심은 **이름이 의도를 표현**하는 것이다. `fetch('/api/users/1')`처럼 URL을 노출하는 대신, `getUserById(id)` 혹은 `fetchCurrentUserProfile()`처럼 **도메인 언어**로 함수를 명명하면, 호출하는 쪽에서는 네트워크 세부 사항을 알지 못해도 의미를 파악할 수 있다. 이는 코드베이스를 하나의 어휘 체계로 만들어 팀 전체의 가독성을 높인다. [github](https://github.com/Ubermensch0608/organize-api-requests)

좋은 API 함수 이름의 조건:

- 동사 + 명사 조합으로 행위를 명확히 표현한다 (`createOrder`, `deleteComment`)
- HTTP 메서드(GET, POST)를 이름에 직접 노출하지 않는다
- 비즈니스 컨텍스트 기준으로 명명한다 (`submitPayment` > `postPaymentData`)
- 모듈 접두사나 네임스페이스로 소속을 나타낼 수 있다 (`userApi.getProfile`)

## 분리가 가져오는 이점

### 단일 수정 지점

API 스펙이 변경될 때, 해당 함수 하나만 수정하면 이를 사용하는 모든 컴포넌트에 변경이 반영된다. URL, 파라미터 구조, 응답 형태가 바뀌어도 컴포넌트 코드는 건드릴 필요가 없다..

### 독립적인 테스트

API 함수가 컴포넌트와 분리되어 있으면, UI를 렌더링하지 않고도 네트워크 로직만 별도로 단위 테스트할 수 있다. 반대로 컴포넌트 테스트에서는 API 함수를 Mock으로 교체해 UI 동작에만 집중할 수 있다..

### 어댑터 패턴과의 결합

라이브러리 의존성을 격리하기 위해 어댑터 패턴과 결합하면, 향후 HTTP 클라이언트나 상태 관리 라이브러리가 교체되어도 도메인 API 계층의 코드를 보존할 수 있다. 이는 장기적인 기술 부채를 줄이는 핵심 전략이다.

### 협업 효율 향상

팀에서 API 명세서와 프론트엔드 함수 목록을 1:1로 대응시킬 수 있어, 백엔드·프론트엔드 간 커뮤니케이션 비용이 감소한다. 새로운 팀원도 `api/` 디렉토리를 보는 것만으로 시스템이 어떤 리소스를 다루는지 파악할 수 있다.

---

## 설계 시 고려 사항

분리된 API 함수를 어떤 계층에서 호출하는지도 중요한 설계 결정이다. 컴포넌트가 직접 호출하는 방식, 커스텀 훅을 통해 간접 호출하는 방식, 상태 관리 스토어를 통해 호출하는 방식 각각은 트레이드오프가 다르다. 어떤 방식을 선택하든 **컴포넌트는 API 함수의 내부를 모르는 상태를 유지**하는 것이 원칙이다.

또한 API 함수는 순수하게 네트워크 통신과 응답 변환만 담당해야 하며, 전역 상태 갱신·에러 알림 표시·라우팅 전환 같은 **부수 효과(side effect)는 호출하는 쪽의 책임**으로 남겨두는 것이 응집도를 높이는 방향이다.

---

## 정리

API 호출 로직의 함수 분리는 "코드를 모아두는 것"이 아니라, **책임의 경계를 명확히 긋는 설계 행위**다. 컴포넌트는 화면을 그리는 일에만, API 함수는 데이터를 가져오는 일에만 집중하게 함으로써, 코드베이스는 변경에 강하고 읽기 쉬운 구조를 갖추게 된다.
