# 프론트엔드 API 레이어의 DTO 경계 설계

프론트엔드 API 레이어에서 DTO(Data Transfer Object)를 활용한 데이터 변환은, 외부 시스템의 변경이 UI 내부 로직에 직접 영향을 미치지 못하도록 **명확한 경계(Boundary)** 를 만드는 아키텍처 전략이다.

---

## 왜 경계가 필요한가

API의 응답 데이터를 컴포넌트에 직접 전달하면, UI는 서버 스키마에 강하게 결합(tight coupling)된다. 백엔드가 필드명을 바꾸거나 구조를 변경할 때마다 해당 데이터를 사용하는 모든 컴포넌트, 훅, 유틸리티 함수에 수정이 파급된다. 이것이 **DTO drift** 문제로, 코드 변경의 영향 범위를 예측하기 어렵게 만든다.

API 레이어에 변환 로직을 집중시키면, 수정이 필요한 지점이 한 곳으로 좁혀진다. 외부 계약(서버 API)이 변경되더라도 내부 도메인 모델은 안정적으로 유지된다.

---

## DTO의 역할과 위치

**DTO(Data Transfer Object)** 는 시스템 간 데이터를 전송하기 위한 순수한 데이터 구조체다. 비즈니스 로직을 포함하지 않으며, 오직 데이터의 형태와 계약만을 정의한다. 프론트엔드에서 DTO는 두 가지 방향으로 존재한다.

- **Request DTO** — UI가 만들어낸 데이터를 서버가 기대하는 형태로 변환하기 위한 스키마 (아웃바운드)
- **Response DTO** — 서버로부터 받은 날것의 응답 데이터를 표현하는 스키마 (인바운드)

DTO는 외부와의 계약을 명문화하는 것이므로, 백엔드 API의 실제 형태를 그대로 반영해야 한다. 필드 네이밍 컨벤션(예: `snake_case`)이 프론트엔드 관습(`camelCase`)과 달라도 DTO는 이를 그대로 유지한다.

---

## 도메인 모델과 DTO의 분리

DTO는 **외부와의 계약**이고, **도메인 모델(Domain Entity)** 은 애플리케이션 내부에서 사용하는 데이터의 형태다. 이 둘은 명확히 분리되어야 한다.

| 구분           | DTO                   | 도메인 모델             |
| -------------- | --------------------- | ----------------------- |
| 존재 목적      | 외부와의 데이터 교환  | 내부 비즈니스 로직 표현 |
| 변경 기준      | 서버 API 스펙 변경 시 | 내부 요구사항 변경 시   |
| 위치           | API 레이어            | 도메인/비즈니스 레이어  |
| 로직 포함 여부 | 없음 (순수 데이터)    | 가능 (도메인 규칙)      |
| 네이밍         | 백엔드 관습 따름      | 프론트엔드 관습 따름    |

도메인 모델은 UI가 실제로 소비하는 데이터 구조이며, DTO와 독립적으로 설계된다. 이 분리 덕분에 서버 응답 포맷이 바뀌어도 컴포넌트는 변경할 필요가 없다.

---

## Mapper: 변환 책임의 집중

DTO와 도메인 모델 간의 변환을 담당하는 것이 **Mapper**다. Mapper는 API 레이어의 경계에 위치하며 다음 두 방향의 변환을 수행한다.

- **Inbound (응답 변환)**: `Response DTO → Domain Model` — 서버 응답 데이터를 애플리케이션이 사용하기 적합한 형태로 변환
- **Outbound (요청 변환)**: `Domain Model → Request DTO` — UI의 내부 상태를 서버가 기대하는 형태로 직렬화

Mapper는 변환 로직의 단일 책임 원칙을 구현한다. 날짜 포맷 파싱, `snake_case` → `camelCase` 변환, 중첩 구조 평탄화, 타입 강제 같은 정제 작업이 모두 이 계층에서 처리된다.

---

## API 레이어는 안티-커럽션 레이어

클린 아키텍처와 DDD(Domain-Driven Design) 관점에서 이 구조는 **Anti-Corruption Layer(ACL)** 패턴과 동일하다. 외부 시스템(백엔드 API)의 개념과 언어가 내부 도메인을 오염시키지 않도록 중간에서 번역하는 방파제 역할을 한다.

이 구조는 다음과 같은 의존성 방향을 만든다.

```
[Component] → [Domain Model] ← [Mapper] ← [DTO] ← [External API]
```

컴포넌트는 도메인 모델만 알고, 도메인 모델은 DTO를 모르며, DTO는 외부 API 스펙에만 의존한다. 이 단방향 의존성이 변경 영향 범위를 통제 가능하게 한다.

---

## 경계 설계의 실질적 효과

이 구조를 도입했을 때 얻는 구체적인 이점들이다.

- **변경 격리**: 백엔드 API 필드명·구조 변경 시 Mapper 수정만으로 흡수 가능
- **테스트 용이성**: 도메인 모델은 네트워크 없이 독립적으로 테스트 가능
- **명시적 계약**: DTO가 외부와의 데이터 계약을 코드로 문서화
- **타입 안전성**: DTO와 도메인 모델에 별도 타입을 정의해 런타임 오류 조기 탐지
- **팀 협업**: 프론트엔드-백엔드 계약이 DTO에 명문화되어 커뮤니케이션 비용 감소

특히 모바일이나 장기 운영 프로젝트에서는 DTO에 직접 의존하는 것이 시한폭탄에 가깝다는 점에서, 이 패턴의 도입은 규모가 커질수록 그 가치가 커진다.

---

## 레이어 구조 요약

```
┌─────────────────────────────────────┐
│           UI / Components           │  ← 도메인 모델만 소비
├─────────────────────────────────────┤
│        Domain / Business Layer      │  ← 도메인 모델 정의
├─────────────────────────────────────┤
│   API Layer (Adapter / ACL 역할)    │  ← DTO ↔ 도메인 모델 변환 (Mapper)
├─────────────────────────────────────┤
│       DTO (외부 계약 정의)           │  ← 서버 스펙 그대로 반영
├─────────────────────────────────────┤
│       External API (Server)         │
└─────────────────────────────────────┘
```

각 계층은 바로 아래 계층에만 의존하며, 상위 계층은 하위 계층의 구체적인 구현을 알지 못한다. 이 단방향 의존성이 전체 아키텍처의 안정성을 보장하는 핵심 원칙이다.
