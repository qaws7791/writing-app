# 서버 상태를 길들이다: TanStack Query가 프론트엔드 아키텍처를 바꾸는 방법

프론트엔드 개발을 하다 보면 어느 순간 이런 코드를 반복해서 작성하고 있는 자신을 발견하게 됩니다.

```typescript
const [data, setData] = useState(null)
const [isLoading, setIsLoading] = useState(false)
const [error, setError] = useState(null)

useEffect(() => {
  setIsLoading(true)
  fetchSomething()
    .then(setData)
    .catch(setError)
    .finally(() => setIsLoading(false))
}, [])
```

컴포넌트가 10개라면 이 패턴은 10번 반복됩니다. 문제는 단순한 코드 중복이 아닙니다. 이 방식이 근본적으로 "서버 상태"라는 개념을 제대로 다루고 있지 않다는 것이 핵심입니다.

---

## 클라이언트 상태와 서버 상태는 다르다

React의 `useState`와 `useReducer`는 클라이언트 상태를 관리하기 위해 태어났습니다. 클라이언트 상태란 모달의 열림/닫힘 여부, 폼 입력값, 현재 선택된 탭처럼 **브라우저 안에서만 존재하고 제어 가능한 상태**입니다. 이 상태는 애플리케이션이 진실의 원천(source of truth)입니다.

반면 서버 상태는 성격이 전혀 다릅니다.

- 서버 상태는 **원격에 존재**하며, 우리는 그것의 복사본을 잠시 가져다 쓸 뿐입니다.
- 우리가 데이터를 보고 있는 동안 **다른 누군가가 서버의 데이터를 변경**할 수 있습니다.
- 네트워크 요청은 비용이 크므로, 같은 데이터를 중복으로 요청하지 않도록 **캐싱**이 필요합니다.
- 네트워크 오류 등으로 인해 **자동 재시도**가 필요한 경우가 생깁니다.

`useEffect + useState` 조합은 이 모든 복잡성을 전혀 처리해 주지 않습니다. 개발자가 직접 구현해야 하고, 그 결과물은 매번 조금씩 다르며 버그가 숨어들기 쉽습니다.

TanStack Query(구 React Query)는 바로 이 지점에서 등장합니다. **서버 상태 관리를 위한 전용 도구**로서, 위에서 나열한 문제들을 라이브러리 수준에서 해결합니다.

---

## TanStack Query가 해결하는 것들

### 1. 보일러플레이트의 소멸

`useQuery` 훅 하나는 `isLoading`, `isError`, `data`, `isFetching` 등 서버 상태를 다루는 데 필요한 모든 파생 상태를 자동으로 제공합니다. 컴포넌트는 더 이상 상태를 직접 선언하고 관리하는 책임을 지지 않아도 됩니다.

이는 단순히 코드가 짧아지는 것을 넘어, **관심사의 분리**가 훨씬 명확해진다는 것을 의미합니다. 컴포넌트는 "데이터를 어떻게 가져올 것인가"를 고민하는 대신 "가져온 데이터를 어떻게 표현할 것인가"에만 집중할 수 있게 됩니다.

### 2. 지능적인 캐싱과 자동 동기화

TanStack Query의 핵심은 **쿼리 키(Query Key)** 기반의 캐싱 시스템입니다. `["user", userId]`라는 키로 등록된 쿼리는 애플리케이션 어디에서 호출하든 동일한 캐시를 공유합니다. 사용자 프로필 페이지와 헤더 컴포넌트가 동시에 같은 사용자 데이터를 필요로 한다면, 네트워크 요청은 단 한 번만 발생합니다.

더 나아가, 사용자가 다른 탭에 갔다가 돌아오거나, 네트워크가 재연결될 때 TanStack Query는 **백그라운드에서 조용히 데이터를 최신 상태로 갱신**합니다. 개발자가 별도로 처리하지 않아도, 사용자는 항상 신선한 데이터를 보게 됩니다.

### 3. Mutation과 캐시 무효화의 우아한 처리

데이터를 읽는 것(Query)만큼 중요한 것이 데이터를 변경하는 것(Mutation)입니다. `useMutation` 훅은 비동기 작업의 실행과 그 결과에 따른 후속 처리를 구조화된 방식으로 표현할 수 있게 해줍니다.

특히 `onSuccess` 콜백에서 `queryClient.invalidateQueries()`를 호출하는 패턴은 매우 강력합니다. 댓글을 새로 작성했다면 댓글 목록 쿼리를 무효화(invalidate)하고, TanStack Query가 알아서 최신 데이터를 다시 불러오도록 위임하는 것입니다. **"이 데이터가 바뀌었으니 관련된 것들을 갱신하라"는 의도가 코드에 명확히 드러납니다.**

---

## 기존 레이어드 아키텍처와의 통합

TanStack Query의 또 다른 강점은 특정 아키텍처를 강제하지 않는다는 점입니다. 도메인(Domain), 애플리케이션(Application), 인프라(Infrastructure) 레이어로 구성된 구조에 자연스럽게 스며듭니다.

역할 분담을 정리하면 다음과 같습니다.

| 레이어                               | 역할                                                |
| ------------------------------------ | --------------------------------------------------- |
| **인프라 (Infrastructure)**          | 실제 API 호출 함수 (`fetch`, `axios` 등)            |
| **애플리케이션 - Query/Mutation 훅** | 인프라 함수를 TanStack Query로 감싸 캐싱·상태 제공  |
| **애플리케이션 - 유즈케이스 훅**     | 도메인 규칙 검증 후 Mutation 훅 조합·오케스트레이션 |
| **컴포넌트 (Presentation)**          | 유즈케이스 훅을 호출해 UI 렌더링에만 집중           |

이 구조에서 TanStack Query는 인프라와 애플리케이션 레이어 사이의 **서버 상태 관리 계층**이라는 명확한 자리를 차지합니다. 도메인 로직은 TanStack Query를 전혀 알 필요가 없고, 컴포넌트는 API 호출 방식을 전혀 알 필요가 없습니다. 각 레이어의 관심사가 이전보다 훨씬 선명하게 구분됩니다.

---

## 무엇이 달라지는가

TanStack Query 도입 전후를 비교하면, 코드의 양이 줄어드는 것보다 더 중요한 변화가 생깁니다.

도입 이전에는 서버 상태 관련 로직이 컴포넌트 곳곳에 산재해 있었습니다. 로딩 처리가 빠진 컴포넌트, 에러를 무시하는 컴포넌트, 캐싱 없이 매번 API를 호출하는 컴포넌트가 혼재합니다. 이는 일관성 없는 사용자 경험으로 이어집니다.

도입 이후에는 서버 상태에 관한 모든 것이 쿼리/뮤테이션 훅으로 캡슐화됩니다. 캐싱, 로딩 상태, 에러 처리, 재시도, 백그라운드 갱신은 라이브러리가 일관되게 처리합니다. **개발자는 "어떻게 가져올 것인가"가 아니라 "무엇을 가져올 것인가"와 "그것으로 무엇을 할 것인가"에만 집중하게 됩니다.**

서버 상태는 클라이언트 상태와 다르며, 그것을 다루는 도구도 달라야 합니다. TanStack Query는 그 명제를 가장 설득력 있게 증명하는 라이브러리입니다.
