# 프론트엔드 HTTP 클라이언트 중앙화

## 개요

현대 프론트엔드 애플리케이션은 수십 개 이상의 API 엔드포인트와 통신한다. 이 요청들이
컴포넌트나 모듈 전반에 산재해 있을 경우, 인증 처리·에러 핸들링·로깅 같은 공통 로직이
중복되고 일관성이 무너진다. **HTTP 클라이언트 중앙화**는 이 모든 네트워크 통신을
단일 레이어로 수렴시켜, 애플리케이션 전체에서 균일한 요청/응답 흐름을 보장하는
아키텍처 전략이다.

---

## 왜 중앙화가 필요한가

### 분산된 HTTP 호출의 문제

컴포넌트마다 개별적으로 fetch 또는 HTTP 라이브러리를 직접 호출하면 다음 문제가 발생한다.

- **중복 로직**: 인증 토큰 주입, 에러 처리, 타임아웃 설정이 호출마다 반복된다.
- **일관성 붕괴**: 담당자마다 에러 처리 방식이 달라져 예측 불가능한 동작이 생긴다.
- **유지보수 취약성**: 기반 HTTP 라이브러리를 교체하거나 베이스 URL을 변경할 때
  전체 코드베이스를 수정해야 한다.
- **테스트 어려움**: 네트워크 로직이 UI 로직과 뒤섞여 단위 테스트가 복잡해진다.

### 중앙화가 해결하는 것

중앙화된 HTTP 클라이언트는 네트워크 계층을 하나의 진입점(single entry point)으로
응집시킨다. 애플리케이션의 나머지 코드는 "어떻게 요청을 보내는가"가 아니라
"무엇을 요청하는가"에만 집중할 수 있게 된다.

---

## 핵심 개념

### 단일 진입점 (Single Entry Point)

모든 HTTP 요청은 하나의 클라이언트 인스턴스 혹은 API 레이어를 통해 발생한다.
이 진입점은 기반 HTTP 라이브러리(Fetch API, Axios, ky 등)의 구체적인 사용
방식을 캡슐화하며, 외부에는 추상화된 인터페이스만 노출한다.

단일 진입점의 역할:

- 베이스 URL, 타임아웃, 공통 헤더 등 전역 설정의 일원화
- 요청·응답 직렬화/역직렬화 표준화
- HTTP 라이브러리 교체 시 영향 범위를 단일 레이어로 제한

### 횡단 관심사 (Cross-Cutting Concerns)

횡단 관심사란 비즈니스 도메인과 무관하게 모든 요청에 공통으로 적용되어야 하는
관심사다. HTTP 클라이언트 중앙화의 핵심 가치는 이 관심사들을 한 곳에서 선언적으로
관리하는 데 있다.

| 관심사          | 설명                                                              |
| --------------- | ----------------------------------------------------------------- |
| **인증**        | Authorization 헤더, Bearer 토큰, 세션 쿠키 자동 주입              |
| **에러 핸들링** | 4xx/5xx 응답의 일관된 에러 변환 및 전파                           |
| **로깅**        | 요청 URL, 메서드, 응답 시간, 상태 코드 기록                       |
| **재시도**      | 네트워크 오류 시 지수 백오프(exponential backoff) 기반 재시도     |
| **캐싱**        | 동일 요청의 응답을 일정 시간 저장하여 불필요한 네트워크 호출 방지 |
| **로딩 상태**   | 전역 로딩 인디케이터 제어                                         |
| **요청 취소**   | 컴포넌트 언마운트 시 진행 중인 요청 자동 중단                     |

### 인터셉터 패턴 (Interceptor Pattern)

인터셉터는 요청이 서버로 전송되기 전, 또는 응답이 호출자에게 전달되기 전에
개입하는 미들웨어다. 중앙화된 클라이언트에서 인터셉터는 횡단 관심사를 선언적으로
분리하는 핵심 메커니즘이다.

```

요청 흐름:

[컴포넌트] → [API 레이어] → [인증 인터셉터] → [로깅 인터셉터] → [서버]
                                                                    ↓
[컴포넌트] ← [API 레이어] ← [에러 인터셉터] ← [응답 변환 인터셉터] ←

```

인터셉터는 **책임 연쇄 패턴(Chain of Responsibility)** 을 따른다.
각 인터셉터는 하나의 책임만 갖고, 다음 인터셉터로 요청을 위임하거나
처리를 중단할 수 있다.

### 추상화와 디커플링 (Abstraction & Decoupling)

중앙화된 클라이언트는 기반 HTTP 라이브러리와 애플리케이션 코드 사이의
어댑터(Adapter) 역할을 한다. 이 어댑터 계층 덕분에:

- 기반 라이브러리(Axios → Fetch API 등)를 교체해도 애플리케이션 코드는 무변경
- 응답 스키마가 변경되었을 때 어댑터 레이어만 수정
- 테스트 환경에서 실제 네트워크 없이 클라이언트 전체를 Mock으로 교체 가능

## 아키텍처 계층 내 위치

HTTP 클라이언트 중앙화는 일반적으로 **데이터 접근 레이어(Data Access Layer)**
에 위치하며, 프레젠테이션 레이어(UI)와 비즈니스 로직 레이어 모두에서 분리된다.

```

┌─────────────────────────────────┐
│         프레젠테이션 레이어        │  컴포넌트, 페이지, UI 상태
├─────────────────────────────────┤
│         비즈니스 로직 레이어       │  상태 관리, 도메인 규칙
├─────────────────────────────────┤
│       API 서비스 레이어           │  도메인별 API 함수 (users, products 등)
├─────────────────────────────────┤
│  ★ HTTP 클라이언트 중앙화 레이어  │  공통 설정, 인터셉터, 에러 변환
└─────────────────────────────────┘

```

Feature-Sliced Design(FSD) 같은 프론트엔드 아키텍처 방법론에서는 이 레이어를
`shared/api`에 위치시키는 것을 권장한다.

## 관련 아키텍처 패턴

### Repository Pattern

Repository 패턴은 HTTP 클라이언트 중앙화 위에 도메인 언어를 덧씌운다.
컴포넌트는 `fetch('/api/users')` 대신 `userRepository.findAll()` 을 호출한다.
이로써 도메인 로직과 네트워크 프로토콜이 완전히 분리된다.

### Backend for Frontend (BFF)

BFF는 HTTP 클라이언트 중앙화의 서버 측 버전이다. 클라이언트별(Web, iOS, Android)
로 전용 백엔드를 두어 각 클라이언트가 필요한 형태의 데이터를 받도록 한다.
프론트엔드의 중앙화된 클라이언트와 BFF를 함께 사용하면, 클라이언트와 서버 양쪽의
관심사를 모두 깔끔하게 분리할 수 있다.

## 트레이드오프와 고려사항

중앙화가 항상 이상적인 해결책은 아니다. 다음 사항을 고려해야 한다.

### 장점

- 공통 로직의 단일 수정 지점(Single Point of Change)
- 네트워크 계층의 예측 가능성 향상
- 라이브러리 교체·마이그레이션 비용 최소화
- API 계약(contract) 변경 시 영향 범위 제한

### 주의점

- **인터셉터 과부하**: 인터셉터가 너무 많아지면 요청마다 순차 처리로 인한
  성능 오버헤드와 디버깅 복잡도가 증가한다.
- **숨겨진 의존성**: 인터셉터가 많을수록 요청에 어떤 변환이 적용되는지
  추적하기 어려워진다. 이를 방지하려면 인터셉터 목록을 명시적으로 선언하고
  문서화해야 한다.
- **과도한 추상화**: 소규모 프로젝트에서 복잡한 중앙화 레이어를 구축하면
  오히려 유지보수 부담이 늘어날 수 있다. 프로젝트 규모에 맞는 수준의
  추상화를 선택하는 것이 중요하다.

## 정리

HTTP 클라이언트 중앙화는 단순한 유틸리티 함수를 만드는 것이 아니라,
**네트워크 계층을 애플리케이션의 명확한 아키텍처 경계로 격리**하는 설계 결정이다.
이 경계가 잘 정의될수록 비즈니스 로직과 UI는 네트워크의 세부사항에 결합되지 않고,
독립적으로 진화할 수 있다.
