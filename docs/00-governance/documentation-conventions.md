# Documentation Conventions

## 목적

문서는 탐색이 쉬워야 하고, 구현 판단에 직접 도움이 되어야 하며, 오래된 내용과 현재 기준을 쉽게 구분할 수 있어야 합니다.

## 파일과 디렉토리 규칙

- 기준 문서 파일명은 영문 `kebab-case`를 사용합니다.
- 각 상위 디렉토리에는 `README.md`를 둡니다.
- 번호가 붙은 디렉토리는 탐색 순서이자 정보 구조입니다.
- 기존 자료를 보존하기 위한 `legacy-*` 파일과 디렉토리는 예외적으로 원본 이름을 유지할 수 있습니다.

## 문서 상태

- `canonical`: 현재 제품과 구현의 기준
- `reference`: 참고용 자료
- `archive`: 더 이상 기준이 아닌 과거 문서

## 서술 규칙

- 구현된 사실과 계획 중인 사항을 분리해서 적습니다.
- 추상적인 비전보다 의사결정에 필요한 구체성을 우선합니다.
- 화면 문서는 목적, 사용자 행동, 핵심 구성, 제외 범위를 함께 적습니다.
- 기술 문서는 책임 경계와 현재 스택을 먼저 적고, 이후 계획을 별도로 적습니다.

## 갱신 트리거

- 정보 구조 변경
- 앱 또는 패키지 책임 변경
- 사용자 흐름 변경
- 운영 정책 변경
- 보안 또는 릴리스 절차 변경
