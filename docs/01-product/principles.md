## 6대 설계 원칙

### 1. 여정 우선 (Journey First)

모든 화면과 기능은 여정(Journey)을 중심으로 설계한다. 여정은 플랫폼의 핵심(Core) 기능이며, 글쓰기는 여정의 연습 공간(Secondary), 글감은 글쓰기를 돕는 보조 도구(Tertiary)다. 이 위계가 탭 구조, 화면 배치, UX 우선순위 전반에 반영된다.

### 2. 사고 우선 (Thinking First)

모든 기능 설계의 기준은 "이것이 사용자의 사고를 자극하는가?"이다. 글을 쓰는 것이 목적이 아니라, 글쓰기를 통해 스스로 생각하는 힘을 기르는 것이 목적이다.

### 3. 점진적 노출 (Progressive Disclosure)

한 번에 모든 것을 보여주지 않는다. 피드백도, 학습 단계도, 기능도 사용자의 수준에 맞춰 점진적으로 노출한다. 입문 수준에서는 강점 피드백 비율을 높이고, 상급으로 갈수록 구조 전체에 대한 비판적 피드백을 제공한다.

### 4. 수정은 성장이다 (Revision = Growth)

초고의 완벽함을 추구하지 않는다. 초안 → AI 피드백 → 수정 → 재제출의 반복 루프를 핵심 경험으로 설계하며, 수정 과정 자체를 긍정적 경험으로 만든다. 수정 전후를 비교하여 개선된 부분을 명시적으로 인정한다.

### 5. 간결함 (Simplicity)

복잡함보다 간결함을 추구한다. 1인 개발 사이드 프로젝트의 제약을 인정하고, 글감 선택 → 글쓰기 → AI 피드백의 핵심 루프에 집중한다.

### 6. 모바일 네이티브 (Mobile-First)

엄지 최적화 UX를 기본으로 설계한다. 세션당 10~15분 완결형 구조, 키보드 대응(visualViewport API, `height: 100dvh`), 하단 탭 바 기반 네비게이션을 원칙으로 한다.

## AI 통합 원칙

### AI의 역할: 소크라테스식 코칭

AI는 완성된 텍스트를 생성하지 않는다. 대신 다음의 역할만을 수행한다.

| 역할       | 설명                           |
| ---------- | ------------------------------ |
| **진단자** | 글의 구조적·논리적 특성을 분석 |
| **질문자** | 사고를 확장하는 질문 제시      |
| **격려자** | 개선된 부분을 구체적으로 칭찬  |
| **안내자** | 다음 단계를 위한 힌트 제공     |

### 피드백 구조

AI 피드백은 다음 3가지 카테고리로만 제공한다.

- **강점 (1~2개):** 구체적으로 잘된 부분과 이유
- **개선점 (1~2개):** 문제 지적 + 방향 힌트 (답을 주지 않음)
- **사고 촉발 질문 (1개):** 글의 깊이를 확장하는 메타인지 질문

### 수준별 피드백 전략

| 수준 | 강점 : 개선점 비율 | 피드백 깊이      |
| ---- | ------------------ | ---------------- |
| 입문 | 2 : 1              | 문장/문단 단위   |
| 중급 | 1 : 1              | 논리 구조 단위   |
| 상급 | 1 : 2              | 에세이 전체 단위 |

### AI가 하지 않는 것

- 완성된 문장이나 문단을 생성하지 않는다.
- 대체 표현을 직접 제시하지 않는다.
- 맞춤법·문법 교정을 자동으로 적용하지 않는다 (지적은 하되, 수정은 사용자의 몫).
- 글의 주제나 방향을 결정하지 않는다.
- AI 제안을 자동으로 반영하지 않고, 사용자가 선택한다.

## 비범위 (이 플랫폼이 아닌 것)

- 책이나 강의를 대체하는 학습 플랫폼이 아니다.
- 노션, 옵시디언 같은 생산성 기록 앱이 아니다.
- 브런치, 문피아 같은 출판·수익화 플랫폼이 아니다.
- 일기 앱 또는 감정 치유 앱이 아니다.
