## 배포 목표

- 사용자가 글을 쓰는 동안 배포로 인해 저장이 끊기지 않아야 한다.
- 웹과 API는 서로 독립적으로 배포할 수 있어야 한다.
- 장애가 발생했을 때 빠르게 이전 안정 버전으로 되돌릴 수 있어야 한다.
- 프리뷰, 스테이징, 운영 환경의 차이를 최소화해야 한다.

## 배포 단위

- `apps/web`: 사용자에게 노출되는 Next.js 애플리케이션
- `apps/api`: 인증, 글쓰기, 글감, AI 보조 요청을 처리하는 Hono API
- 오브젝트 스토리지: 내보내기 파일, 첨부 자산, 공개용 파생 파일 저장
- 관계형 데이터베이스: 사용자, 세션, 글, 버전, 공개 설정, 운영 이벤트 저장
- 관측 스택: 로그, 메트릭, 트레이스를 수집하고 알림을 발송하는 운영 계층

`packages/ui`와 설정 패키지는 독립 배포 대상이 아니라 앱 빌드에 포함되는 공유 코드로 다룬다.

## 환경 전략

### 로컬

- 개발자는 `web`과 `api`를 분리 실행한다.
- 로컬 환경에서도 인증, 자동 저장, AI 목업 흐름을 재현할 수 있어야 한다.
- 파일 저장은 운영과 같은 인터페이스를 사용하되, 로컬 에뮬레이터나 개발 전용 버킷을 사용한다.

### 스테이징

- 운영과 동일한 배포 구조를 사용한다.
- 새 인증 정책, 저장 포맷, 마이그레이션은 먼저 스테이징에서 검증한다.
- QA와 제품 검토는 스테이징 URL을 기준으로 진행한다.

### 운영

- 웹과 API는 별도 배포 파이프라인을 가진다.
- 데이터베이스와 스토리지는 다중 가용영역 또는 동등한 고가용성 구성을 기본으로 한다.
- 운영 비밀값은 배포 플랫폼에 저장하고 저장소에는 커밋하지 않는다.

## 릴리스 원칙

- 기본 전략은 작은 변경을 자주 배포하는 방식이다.
- 웹은 프리뷰 배포를 우선 제공하고, 운영 반영은 검토 후 승격한다.
- API는 스테이징 검증 후 운영에 반영한다.
- 스키마 변경은 항상 하위 호환 순서로 배포한다.
  1. 새 스키마 추가
  2. 애플리케이션 이중 읽기 또는 이중 쓰기 반영
  3. 데이터 백필
  4. 구 스키마 제거

## 데이터베이스와 마이그레이션

- 글쓰기 데이터는 손실 비용이 크므로 파괴적 마이그레이션을 즉시 실행하지 않는다.
- 자동 저장과 버전 기록 테이블은 마이그레이션 중에도 읽기 가능해야 한다.
- 운영 마이그레이션은 배포와 분리된 명시적 단계로 실행한다.
- 실패 시 즉시 되돌릴 수 있도록 롤포워드와 롤백 절차를 함께 문서화한다.

## 설정과 비밀 관리

- 환경 변수는 앱별로 분리한다.
- 인증 시크릿, 세션 키, AI 제공자 키, 스토리지 서명 키는 모두 회전 가능해야 한다.
- 브라우저에 노출되는 설정과 서버 전용 설정을 명확히 구분한다.
- 비밀값 변경은 무중단 회전을 우선하고, 불가피할 때만 짧은 유지보수 창을 사용한다.

## 롤백 전략

- 웹은 직전 안정 빌드로 즉시 되돌릴 수 있어야 한다.
- API는 새 배포가 저장, 인증, 공개 흐름에 영향을 주면 즉시 이전 이미지로 되돌린다.
- 데이터 스키마가 이미 전진한 경우에는 애플리케이션을 호환 모드로 유지한 채 롤포워드를 우선 검토한다.

## 운영 체크리스트

- 배포 전: 스테이징에서 로그인, 자동 저장, AI 제안, 내보내기 흐름 확인
- 배포 직후: 오류율, 저장 성공률, 주요 응답 시간, 로그인 실패율 확인
- 배포 후 30분: 사용자 영향 알림 유무와 롤백 필요성 재판단
