## 저장 원칙

- 글의 정본은 데이터베이스에 저장하고, 파일 저장소는 보조 저장소로 사용한다.
- 로컬 디스크는 일시적 처리 공간일 뿐 영구 저장소로 사용하지 않는다.
- 공개 범위가 다른 파일은 같은 버킷 경로를 공유하지 않는다.
- 파일 접근 권한은 URL 자체가 아니라 서버가 발급한 정책으로 통제한다.
- 사용자는 임의의 파일을 별도로 업로드할 수 없고, 에디터 안에 삽입할 이미지 파일만 업로드할 수 있다.
- 에세이 읽기를 방해하는 움직이는 이미지 파일은 허용하지 않는다.

## 저장 대상

### 파생 파일

- 사용자가 내보낸 `txt`, `md` 파일
- 향후 추가될 PDF, 인쇄용 파생본
- 공개용 읽기 전용 렌더링 파일

### 사용자 자산

- 커버 이미지
- 글에 삽입되는 정적 이미지 파일
- 신고나 운영 검토에 필요한 증빙 파일

현재 단계에서 일반 첨부 파일 업로드 기능은 제공하지 않는다.

### 시스템 자산

- 썸네일, 변환본, 바이러스 검사 결과 같은 비동기 처리 산출물

## 파일과 데이터의 역할 분리

- 제목, 본문, 버전 이력, 공개 상태는 데이터베이스에서 관리한다.
- 파일 저장소에는 원문 복사본을 무분별하게 쌓지 않는다.
- 버전 기록은 파일 버전보다 도메인 버전으로 관리한다.
- 업로드된 이미지는 독립 다운로드용 자산이 아니라 에디터 본문 안에서 참조되는 삽입 자산으로 다룬다.
- 파일 메타데이터만 데이터베이스에 저장한다.
  - 소유자
  - MIME 타입
  - 크기
  - 해시
  - 저장 경로
  - 공개 범위

## 경로 설계

- 경로는 사용자가 추측하기 쉬운 순번 대신 난수 기반 식별자를 사용한다.
- 버킷 또는 프리픽스는 최소한 아래 범위로 분리한다.
  - 비공개 사용자 자산
  - 제한 공유 자산
  - 공개 파생 자산
  - 운영 전용 검역 자산

예시 구조:

```text
private/users/{user-id}/writings/{writing-id}/exports/{file-id}.md
private/users/{user-id}/assets/{file-id}
shared/writings/{share-id}/preview/{file-id}.html
public/writings/{publication-id}/cover/{file-id}
quarantine/uploads/{file-id}
```

## 업로드와 다운로드 정책

- 업로드는 API가 발급한 제한 시간 서명 URL 또는 동등한 방식으로 처리한다.
- 업로드 가능 대상은 에디터 삽입용 이미지 파일만 허용한다.
- 업로드 최대 크기는 파일당 5MB다.
- GIF와 기타 움직이는 이미지 형식은 허용하지 않는다.
- 다운로드는 공개 범위에 따라 다르게 제공한다.
  - 비공개 파일: 인증된 소유자만 접근
  - 제한 공유 파일: 만료 가능 링크 또는 공유 토큰으로 접근
  - 공개 파일: 캐시 가능한 읽기 전용 URL 제공
- 내보내기 파일은 요청 시 생성하거나 짧은 수명으로 캐시한다.

## 보안 정책

- 모든 업로드 파일은 MIME 타입과 실제 바이너리 타입을 함께 검사한다.
- 업로드 허용 형식은 정적 이미지 형식으로 제한한다.
- GIF는 확장자와 MIME 타입 양쪽에서 차단한다.
- 이미지 파일은 악성 코드 검사 또는 동등한 검역 단계를 통과해야 한다.
- HTML 기반 파생 파일은 스크립트 삽입을 허용하지 않는다.
- 파일명은 사용자 입력을 그대로 경로로 사용하지 않는다.
- 크기 제한 초과나 허용되지 않은 형식은 업로드 시작 전과 저장 직전 두 단계에서 모두 검증한다.

## 수명 주기 관리

- 임시 내보내기 파일은 일정 시간이 지나면 자동 삭제한다.
- 삭제된 글에 연결된 비공개 파일은 복구 가능 기간 이후 정리한다.
- 공개 취소된 자산은 CDN 캐시 무효화와 저장소 비공개 전환을 함께 처리한다.
- 고아 파일을 주기적으로 탐지해 정리한다.

## 비용과 성능

- 원본과 파생본의 중복 저장을 최소화한다.
- 자주 조회되는 공개 파일만 CDN 캐시 대상으로 둔다.
- 단일 파일 최대 5MB이므로 초기 단계에서는 단순 업로드 경로를 우선한다.
- 파일 접근 로그를 남겨 오남용과 비용 급증을 감지한다.

## 운영 기준

- 파일 삭제는 실제 삭제 전 메타데이터 상태를 먼저 변경한다.
- 저장 실패가 발생해도 글 본문 저장은 독립적으로 성공할 수 있어야 한다.
- 파일 전략은 향후 모바일 앱이나 외부 공유 기능이 추가되어도 같은 권한 모델을 재사용할 수 있어야 한다.
- 에디터 이미지 업로드 정책이 바뀌더라도 기존 글의 이미지 참조는 깨지지 않아야 한다.
