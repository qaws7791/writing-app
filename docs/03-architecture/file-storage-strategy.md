## 저장 원칙

- 글의 정본(제목, 본문, 버전 이력, 진행 상태)은 데이터베이스에 저장하고, 파일 저장소는 보조 저장소로 사용한다.
- 로컬 디스크는 일시적 처리 공간일 뿐 영구 저장소로 사용하지 않는다.
- 파일 접근 권한은 URL 자체가 아니라 서버가 발급한 정책으로 통제한다.
- 현재 단계에서 사용자가 직접 파일을 업로드하는 기능은 제공하지 않는다.

## 저장 대상

### 시스템 자산

- 여정(Journey) 썸네일 이미지
- 글감(Writing Prompt) 관련 이미지 (선택)
- 온보딩, 마케팅 등 서비스 운영에 필요한 정적 자산

### 파생 파일

- 사용자가 "데이터 내보내기"를 요청할 때 생성되는 마크다운 파일 패키지
- 향후 추가될 수 있는 PDF, 인쇄용 파생본

## 파일과 데이터의 역할 분리

- 제목, 본문, 버전 이력, 여정 진행 상태는 데이터베이스에서 관리한다.
- 파일 저장소에는 원문 복사본을 무분별하게 쌓지 않는다.
- 버전 기록은 파일 버전보다 도메인 버전으로 관리한다.

## 경로 설계

- 경로는 사용자가 추측하기 쉬운 순번 대신 난수 기반 식별자를 사용한다.
- 버킷 또는 프리픽스는 최소한 아래 범위로 분리한다.
  - 시스템 자산 (여정 썸네일 등)
  - 사용자 내보내기 파일

예시 구조:

```text
system/journeys/{journey-id}/thumbnail/{file-id}
system/prompts/{prompt-id}/image/{file-id}
exports/users/{user-id}/{export-id}.zip
```

## 다운로드 정책

- 시스템 자산(여정 썸네일 등)은 캐시 가능한 읽기 전용 URL로 제공한다.
- 내보내기 파일은 인증된 소유자만 접근하며, 짧은 수명의 서명 URL로 제공한다.
- 내보내기 파일은 요청 시 생성하거나 짧은 수명으로 캐시한다.

## 보안 정책

- 파일명은 사용자 입력을 그대로 경로로 사용하지 않는다.
- 시스템 자산 업로드는 운영 권한으로만 가능하다.
- HTML 기반 파생 파일은 스크립트 삽입을 허용하지 않는다.

## 수명 주기 관리

- 임시 내보내기 파일은 일정 시간이 지나면 자동 삭제한다.
- 고아 파일을 주기적으로 탐지해 정리한다.

## 비용과 성능

- 원본과 파생본의 중복 저장을 최소화한다.
- 자주 조회되는 시스템 자산만 CDN 캐시 대상으로 둔다.
- 파일 접근 로그를 남겨 오남용과 비용 급증을 감지한다.

## 운영 기준

- 파일 삭제는 실제 삭제 전 메타데이터 상태를 먼저 변경한다.
- 저장 실패가 발생해도 글 본문 저장은 독립적으로 성공할 수 있어야 한다.
