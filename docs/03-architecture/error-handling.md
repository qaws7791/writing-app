## 오류 처리 원칙

- 오류 메시지는 사용자를 탓하지 않는다.
- 실패보다 복구 경로를 먼저 보여준다.
- 글쓰기 도중에는 본문 보존이 가장 우선이다.
- 같은 종류의 오류는 웹과 API에서 같은 코드 체계를 사용한다.
- 비즈니스 계층에서는 오류를 예외보다 값으로 다룬다.
- HTTP 경계에서만 Problem Details 응답으로 변환한다.

## 오류 분류

### 사용자 입력 오류

- 필수값 누락
- 길이 제한 초과
- 공개 범위나 공유 설정의 잘못된 조합
- 허용되지 않은 파일 형식 또는 업로드 크기 초과

이 경우 사용자에게 즉시 수정 가능한 문구를 보여준다.

### 인증 및 권한 오류

- 로그인 만료
- CSRF 검증 실패
- 타인의 글 또는 파일 접근 시도

이 경우 보안 정보는 최소화하고 재인증 또는 접근 불가 상태를 안내한다.

### 충돌 오류

- 오래된 버전 위에 저장 시도
- 이미 삭제되었거나 공개 상태가 바뀐 리소스 수정

이 경우 사용자의 현재 초안을 보존한 채 최신 상태와 비교할 수 있어야 한다.

### 외부 의존성 오류

- AI 제공자 지연
- 스토리지 업로드 실패
- 데이터베이스 연결 실패

핵심 흐름을 막지 않는 범위에서 기능 축소 모드로 전환한다.

### 내부 서버 오류

- 예기치 않은 예외
- 데이터 정합성 오류
- 타임아웃

사용자에게는 일반화된 안내를 제공하고, 운영에는 상세 원인을 남긴다.

## 내부 오류 흐름 기준

- `packages/core`는 예외를 기본 제어 흐름으로 사용하지 않는다.
- use case는 `neverthrow`의 `Result`, `ResultAsync` 같은 실패 값을 우선 사용한다.
- `packages/db`, `packages/storage`, `packages/ai`는 외부 오류를 의미 있는 포트 오류나 실패 값으로 변환한다.
- `apps/api` presenter가 이 값을 Problem Details로 매핑한다.
- 예상하지 못한 런타임 예외만 최외곽 fallback에서 기록하고 일반화된 500 응답으로 처리한다.

## API 오류 응답 형식

API 오류 응답은 RFC 7807/9457 Problem Details for HTTP APIs를 기반으로 한다.

- 응답 `Content-Type`은 `application/problem+json`을 사용한다.
- 우선은 필수 필드만 응답에 포함한다.
- 내부 예외 메시지는 그대로 노출하지 않는다.

기본 응답 예시는 아래와 같다.

```json
{
  "type": "https://api.example.com/problems/writing-conflict",
  "title": "Writing conflict",
  "status": 409
}
```

현재 단계에서는 아래 필드만 표준 응답으로 사용한다.

- `type`: 문제 유형을 식별하는 URI
- `title`: 사람이 읽을 수 있는 짧은 요약
- `status`: HTTP 상태 코드

`detail`, `instance` 같은 필드는 추후 필요가 분명해질 때 추가한다.

## HTTP 상태 코드 기준

- `400`: 입력 검증 실패
- `401`: 인증 필요 또는 세션 만료
- `403`: 권한 없음
- `404`: 존재하지 않거나 접근할 수 없는 리소스
- `409`: 저장 충돌 또는 상태 충돌
- `422`: 의미적 검증 실패
- `429`: 속도 제한
- `500`: 복구 불가능한 서버 오류
- `503`: 일시적 외부 의존성 장애

## 클라이언트 처리 원칙

- 자동 저장 실패 시 즉시 편집 내용을 지우지 않는다.
- 네트워크 오류는 배너 또는 토스트로 알리고 백그라운드 재시도를 시도한다.
- 인증 오류는 재로그인 흐름으로 연결하되, 복귀 후 작성 중 상태를 복원한다.
- AI 오류는 에디터 전체를 막지 않고 해당 기능만 닫는다.
- 이미지 업로드 오류는 해당 이미지 삽입 흐름만 중단하고 본문 편집은 계속 가능해야 한다.

## 사용자 메시지 원칙

- 좋음: "저장에 잠시 문제가 있습니다. 연결이 복구되면 다시 시도합니다."
- 좋음: "세션이 만료되었습니다. 다시 로그인하면 초안 복원을 계속할 수 있습니다."
- 좋음: "이미지는 5MB 이하의 정적 이미지 파일만 업로드할 수 있습니다."
- 좋음: "움직이는 이미지는 에세이 읽기를 방해해 업로드할 수 없습니다."
- 피해야 함: "알 수 없는 오류가 발생했습니다."
- 피해야 함: "잘못된 요청입니다."

## 운영 처리 원칙

- 모든 서버 오류는 `request_id`와 함께 로그에 남긴다.
- 반복되는 검증 오류는 UX 문제일 수 있으므로 제품 지표로도 본다.
- AI와 스토리지 오류는 외부 제공자 상태와 분리해서 기록한다.
- 장애 대응 후에는 사용자 영향 범위와 데이터 손실 여부를 반드시 확인한다.
- 포트 구현체에서 발생한 오류는 provider 원문을 그대로 노출하지 않고 의미 있는 내부 코드로 정규화한다.

## 폴백 전략

- AI 기능이 실패해도 수동 작성과 저장은 계속 가능해야 한다.
- 파일 내보내기 실패는 글 본문 저장 실패로 취급하지 않는다.
- 제한 공유 기능이 일시 중단되어도 비공개 초안 작성은 유지한다.
