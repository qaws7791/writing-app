## 인증 목표

- 로그인 자체보다 글쓰기 학습 진입의 마찰을 낮춘다.
- 세션은 안전해야 하지만, 세션 진행 중이나 글쓰기 중에 자주 끊겨 내용이 유실되면 안 된다.
- 모바일 브라우저에서도 재인증 흐름이 과도하게 복잡하지 않아야 한다.
- 회원가입 없이 체험 글쓰기가 가능해야 하며, 저장 시점에 가입을 유도한다.

## 인증 수단

### 소셜 로그인

- Google 소셜 로그인
- Kakao 소셜 로그인

두 수단 모두 같은 사용자 계정으로 연결될 수 있어야 하며, 사용자는 나중에 로그인 수단을 추가할 수 있어야 한다.

## better-auth 코어 스키마 기준

현재 인증 구현은 better-auth의 코어 스키마를 기준으로 설계한다.

- `user`: 인증 가능한 기본 사용자 레코드
- `session`: 로그인 세션 레코드
- `account`: 소셜 로그인을 포함한 인증 계정 레코드

이 세 테이블은 인증 인프라의 기준 스키마이며, 글쓰기 도메인 문서에서는 아래처럼 대응시킨다.

- `user` → `User`
- `session` → `Session`
- `account` → `AuthAccount`

제품 문서의 도메인 개념과 DB 스키마 이름이 완전히 같을 필요는 없지만, 매핑은 항상 1:1로 추적 가능해야 한다.

## 세션 모델

- 웹 우선 아키텍처에서는 서버 관리형 세션을 기본으로 한다.
- 브라우저에는 `HttpOnly`, `Secure` 속성을 가진 세션 쿠키만 저장한다.
- 세션 데이터는 서버 또는 중앙 저장소에서 관리한다.
- 세션은 절대 만료 시간과 활동 기반 연장 시간을 함께 가진다.
- better-auth의 `session.token`은 고유해야 하며, 브라우저가 직접 해석하는 토큰이 아니라 서버가 검증하는 세션 식별자로 다룬다.
- `ipAddress`, `userAgent`는 보안 감사와 이상 징후 감지를 위한 메타데이터로 사용한다.

## 세션 수명 정책

- 짧은 비활동 시간 만료로 사용자를 자주 쫓아내지 않는다.
- 장기 로그인은 가능하지만, 민감한 작업 전에는 최근 인증 여부를 다시 확인할 수 있어야 한다.
- 비밀번호 변경, 계정 복구, 의심 로그인 감지 시 모든 활성 세션을 폐기할 수 있어야 한다.
- 공개 기기 사용을 고려해 로그아웃은 명시적이고 즉시 반영되어야 한다.

## 권한 모델

### 사용자 본인만 가능한 작업

- 글 생성, 수정, 삭제
- 여정 참여, 세션 진행, 스텝 응답 제출
- 버전 기록 조회
- 데이터 내보내기
- 프로필 및 온보딩 데이터 수정
- 구독 관리

### 운영 권한

- 신고 검토
- 사용자 데이터 접근 제한
- 감사 이벤트 조회

운영 권한은 일반 사용자 권한과 별도 역할로 분리한다.

## 세션과 글쓰기 보호

- 세션 만료가 임박하면 저장 전에 재인증 또는 세션 연장 신호를 보낸다.
- 자동 저장 요청이 인증 실패로 거절되면, 클라이언트는 로컬 임시 상태를 유지한 채 재시도 흐름으로 전환한다.
- 사용자는 세션 문제로 작성 중인 내용이나 세션 진행 상태를 잃지 않아야 한다.
- 여정 세션 진행 중 앱이 종료되더라도, 마지막 완료된 스텝까지의 상태는 서버에 보존된다.

## API 인증 방식

- 브라우저에서 API 호출 시 기본 인증 수단은 세션 쿠키다.
- 상태 변경 요청에는 CSRF 방어를 포함한다.
- 외부 클라이언트나 향후 모바일 앱이 필요할 경우에만 토큰 기반 인증을 별도 도입한다.
- 같은 계정이라도 브라우저 세션과 외부 토큰은 독립적으로 폐기할 수 있어야 한다.
- better-auth의 `account` 레코드는 `providerId`, `accountId` 조합으로 외부 공급자 계정을 식별한다.

## 온보딩과 인증 흐름

- 앱 최초 실행 시 온보딩(관심사, 글쓰기 수준 선택)은 인증 전에 진행할 수 있다.
- 글감 탐색과 체험 글쓰기는 비인증 상태에서도 가능하다.
- 글 저장 시점에 회원가입을 유도하며, 온보딩 데이터는 가입 완료 시 계정에 연결한다.

## 계정 연결과 복구

- Google 계정과 Kakao 계정은 검증된 소유권 확인 후 연결한다.
- 중복 가입이 발생해도 글과 활동 이력이 분산되지 않도록 병합 절차를 제공한다.
- 계정 복구는 공격 표면이 크므로 단순 편의보다 소유권 검증을 우선한다.
- 비밀번호 해시는 `account.password`에 저장되므로, 사용자 프로필과 인증 비밀은 같은 책임으로 다루지 않는다.
- `emailVerified`는 프로필 완성 여부가 아니라 인증 신뢰 상태로 취급한다.

## 감사와 보안 이벤트

- 로그인 성공, 실패, 로그아웃, 세션 폐기, 비밀번호 변경, 계정 연결은 감사 이벤트로 기록한다.
- 비정상 로그인 패턴은 관측 시스템에서 감지 가능해야 한다.
- 인증 시스템은 사용자 경험을 해치지 않는 범위에서 속도 제한과 이상 징후 차단을 적용한다.
